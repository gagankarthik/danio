Read the engine, not the changelog

This is the actual reconciler.

Most frameworks ask you to trust a compiled blob. Danio ships its source and explains it. Here's the real bailout check — the reason a keystroke re-renders one component instead of two thousand.

src/core/reconciler.js 
// Can we skip re-rendering this component entirely?
function beginWork(fiber) {
  const clean = !fiber.dirty && canReuseProps(fiber)

  if (clean) {
    if (!fiber.childDirty)
      return reuseWholeSubtree(fiber)  // skip the branch
    return bailoutButWalk(fiber)       // a child is dirty
  }

  fiber.dirty = false
  return renderComponent(fiber)          // actually run it
}
1

Two flags, not a diff

A setState marks its own fiber dirty, then drops childDirty breadcrumbs up the parent chain. Rendering just follows the trail.

2

Props compared by identity

canReuseProps checks whether the props are the same object as last render — no deep compare. A parent that bailed never built new ones, so children bail too.

3

Clean branches are free

No dirty child means the entire subtree is handed back by reference. Danio never even walks it. That's the ~200× win — and it's four lines you can read.

Hooks & effects

Title.jsx
import { useState, useEffect } from 'danio'

function Title({ label }) {
  const [n, setN] = useState(0)

  useEffect(() => {
    document.title = `${label}: ${n}`
  }, [label, n])

  return <button onClick={() => setN(n+1)}>{n}</button>
}

Render on the server

Ship HTML first, hydrate second.

Call renderToString in plain Node — no DOM, no mocks — to send real markup for search engines and first paint. Then hydrate adopts those exact nodes on the client and wires up the interactivity, reusing the DOM instead of rebuilding it.

Zero DOM on the server No flash, no rebuild SEO-ready output
server + client
// server.js — runs in Node
import { renderToString } from 'danio/server'
const html = renderToString(<App />)
res.send(`<div id="root">${html}</div>`)

// client.js — runs in the browser
import { hydrate } from 'danio'
hydrate(<App />, document.getElementById('root'))
// same nodes, now interactive.

Reach for Danio when…

  • Bundle size is a hard budget — embedded, widgets, kiosks, low-end devices.
  • You want to understand and audit every line running in your product.
  • It's a controlled internal platform, not a hire-100-people app.
  • You're learning how a framework actually works, hands-on.

Choose something else when…

  • You need streaming SSR or React Server Components right now.
  • Your project leans on a big library ecosystem or a deep hiring pool.
  • You want maximal raw throughput — a signals framework will edge it.
  • An org standard already mandates a specific framework.
